Executive summary

Reactive Wire has the right high-level shape: a headless server, a DOM-free shared engine, thin HA adapters, registry-backed node definitions, and a React Flow editor. The problems are mostly at boundaries—validation, identity, lifecycle, effects, collaboration, and accessibility—not a reason to discard the core thesis.

What is architecturally right

  • Server dependency direction is clean; it does not import frontend code.
  • Editor and server genuinely share one evaluator and value/error semantics.
  • Each built-in node is isolated behind a small NodeDef registry entry.
  • Clock and async-source values are explicit evaluator inputs, which makes tests reproducible.
  • A valid canonical graph evaluates synchronously from one entity/source snapshot; all pin values are materialized before sink intents are built.
  • React Flow, Yjs, Tailwind, and home-assistant-js-websocket are appropriate choices for their respective roles.
  • Backpressure, stale fetch-response rejection, non-Ok sink gating, and reconciliation have meaningful coverage.

What blocks production confidence

  • The server trusts client-supplied node schemas and editor-only type/DAG rules.
  • Ordinary object dictionaries plus reserved IDs permit prototype pollution.
  • String-concatenated identities alias distinct nodes/pins and macro paths.
  • Macro caps apply before expansion, allowing million-node materialization.
  • Effects can overlap, retry forever, never retry, or be consumed before acknowledgment.
  • Normal multi-flow editing has deterministic data-loss reproductions that are currently fixme.
  • The core editor workflow is pointer-only; dialogs and auto-deploy are not keyboard-safe.

Evidence-backed findings

CriticalArchitecture / safetyF-01 — A node named __proto__ mutates Object.prototype100% reproduced

Memory is an ordinary Record<string, NodeMemory>. Stateful nodes read memory[n.id] without an own-property check. Deployment sanitization rejects dangerous object keys, but accepts the same strings as node-ID values.

Evidence: shared/engine/engine-support.ts:12-26, shared/engine/evaluate.ts:223-240,246-265, shared/engine/nodes/edge.ts:10-24, src/server/deploy-validation.ts:80-95.
sanitized true
{"memoryOwn":false,"objectPrototypeSeeded":true,
 "objectPrototypePrevVal":{"type":"bool","v":true,"status":"ok"}}

Impact

A deployed or persisted graph can corrupt process-wide ordinary object behavior, cross-contaminate node memory, and make 24/7 evaluation unpredictable.

Fix

Immediately reject __proto__, prototype, and constructor in every identifier position. Replace runtime dictionaries with Map or null-prototype records and use own-property checks. Add direct-engine and deployment regression tests.

CriticalPerformance / availabilityF-02 — Macro expansion has no cumulative node, edge, or depth budget100% reproduced

Validation allows 1,000 top-level nodes and independently allows 1,000 nodes per macro across 100 macros. Deployer.deploy() then recursively materializes every placement without a post-expansion cap.

Evidence: src/server/deploy-validation.ts:19-23,127-169, shared/engine/expand.ts:65-113,186-233, src/server/runtime.ts:189-205.
accepted graph: 1,000 placements × 1,000-node macro
potential expanded nodes: 1,000,000
nested fan-out can be multiplicative

Impact

An authenticated deploy or persisted auto-deploy document can freeze or exhaust the always-on server before its first evaluation.

Fix

Make expansion budget-aware and abort before materialization. Cap total expanded nodes, edges, pins, placement count, and nesting depth; collision-check the final graph. Keep the last valid live deployment when a replacement fails compilation.

HighArchitecture / actuationF-03 — Server deployment bypasses the advertised typed DAG contract100% reproduced

Type compatibility and cycle validation live in the frontend. Server sanitization checks endpoint strings and node existence, but not registry membership, canonical pin schemas, pin existence/direction, unique pin IDs, single-source inputs, type compatibility, or cycles.

Evidence: frontend/src/canvas/validation.ts:20-71, src/server/deploy-validation.ts:60-137,156-172, shared/engine/evaluate.ts:149-175, shared/engine/nodes/logic.ts:68-73.
accepted: number(1) → sink-light.on(bool)
sink input: { type: "num", v: 1, status: "ok" }
result: light.turn_on(...)

Two wires into one input are also accepted; the last edge silently wins, so reversing edge order changes output and can change actuation.

Fix

Create one shared semantic compiler/validator used by the editor, explicit deploy, auto-deploy, persisted-document load, and tests. Only compiled canonical graphs may reach Deployer.

HighDeterminismF-04 — Delimiter-based IDs alias distinct pins and macro nodes100% reproduced

Pin identity is flattened as ${nodeId}:${pinId}; flow/macro namespaces concatenate /. Neither separator is escaped or rejected.

Evidence: shared/engine/evaluate.ts:149-175,268-275, shared/engine/expand.ts:12-16,92-96,210-218, shared/engine/flow-graphs.ts:16-34, src/server/deploy-validation.ts:60-95.
node "a:b" / pin "c"  = 1
node "a"   / pin "b:c" = 2
both become result key "a:b:c"; observed output was 1

Impact

Array ordering can alter values, health, memory, sink bookkeeping, and macro state isolation.

Fix

Use numeric compiled indices or nested maps/tuples. As an immediate guard, reserve delimiters and collision-check after flow namespacing and macro expansion.

HighEffects / orderingF-05 — Different calls for one sink can execute concurrentlyHigh

The runtime suppresses only an identical in-flight key. A changed call overwrites the one tracked key while the old promise still runs. Rapid on → off → on can issue three concurrent calls; HA completion order, not graph order, can determine the final world state.

Evidence: src/server/runtime.ts:300-312,322-345.

Fix

Give each sink a serialized executor. Reconciling sinks should coalesce to the latest desired correction; transient sinks need a bounded FIFO and explicit overflow policy. Keep transaction order and delivery state observable.

HighEffects / reliabilityF-06 — Delivery and retry policies lose events or hit operational extremesHigh
  • Notify/TTS advances prevVal while merely building an intent, so a failed call consumes the event and is not retried.
  • A failed generic call can retry on every unrelated recompute—including every one-second tick.
  • A failed reconciling call records its attempt before execution and may suppress that context indefinitely.
Evidence: shared/engine/nodes/sink-transient.ts:13-31, src/server/runtime.ts:292-345, test/sink-runtime.test.ts:124-147,184-209.

Impact

Transient messages can disappear; a persistent generic failure can create 86,400+ attempts/logs per day; a one-off HA failure can leave a reconciling target wrong indefinitely.

Fix

Separate “desired,” “enqueued,” “attempted,” and “acknowledged” state. Use bounded exponential backoff with jitter and rate caps. Commit “last delivered” only after acknowledgment for at-least-once transient semantics.

HighLifecycleF-07 — Deployer.stop() does not deactivate the graph100% reproduced

The HA unsubscribe is discarded. stop() clears timers and maps but does not unsubscribe, clear graph, or set actuate=false. A later entity event still runs and may actuate. The current CLI immediately exits, which masks the broken class contract.

Evidence: src/server/runtime.ts:169-180,208-220,273-276.

Fix

Retain and invoke the unsubscribe, null the graph, disable actuation before teardown, abort source requests, and test that post-stop events cannot evaluate or call services.

MediumAtomicity / abstractionF-12 — The NodeDef contract cannot guarantee atomic multi-output updatesHigh

NodeDef.eval(ctx) returns one output pin and is allowed to mutate node memory. The evaluator invokes it once per declared output. Existing stateful built-ins happen to expose one output, but the server accepts noncanonical output lists: a two-output edge produced true for the first output and false for the second because the transition was consumed.

Evidence: shared/engine/node-def.ts:77-96, shared/engine/evaluate.ts:244-272, shared/engine/nodes/edge.ts:10-25.

Fix

Make each node evaluation return all outputs and a proposed next-memory value exactly once per transaction. Commit outputs and memory together only after the full pass. Canonical schema validation is the immediate containment.

MediumServer performanceF-13 — Every event and every second rebuilds and evaluates the entire graphHigh

Every HA callback, fetch completion, and unconditional one-second clock tick rebuilds node/edge indexes, string keys, caches, all pin results, health, sink intents, and another node map—even when the graph has no time nodes or the changed entity is unrelated.

Evidence: src/server/runtime.ts:169-180,273-302, shared/engine/evaluate.ts:146-156,246-303.

The current cost is measurable but not catastrophic for canonical graphs; see the benchmark section. Optimize scheduling and compilation before adopting a new reactive library.

Fix

Compile immutable graph indexes once per deploy. Propagate changed entity/source IDs, mark dirty descendants, and enable clock work only for time-dependent closures. Coalesce source changes into one transaction without collapsing meaningful edge transitions.

MediumHA / feed performanceF-14 — HA callbacks amplify full entity snapshotsHigh

Each HA callback clones two full maps and scans the full merged HA snapshot. Runtime reads allocate a full object. The editor feed serializes every entity for every connected client after a 150 ms throttle.

Evidence: src/ha/apply-entities.ts:37-60, src/ha/real.ts:50-64, src/server/feed.ts:166-172,267-281.

Cost model

With H entities and update rate U, server work is roughly O(H×U); editor bandwidth is O(H×min(U,6.7)×clients).

Fix

Cache a canonical snapshot, propagate changed IDs/deltas through EntityFeed, send one full initial feed then versioned deltas, and let the compiled graph map entity IDs directly to source nodes.

Performance assessment

The current evaluator is not intrinsically slow. The waste comes from when and how often it is invoked, repeated allocations/index construction, whole-entity snapshots, persistence work, and adversarial expansion.

Interpretation

  • A canonical 1,000-node graph at one fixed tick consumes roughly 340 CPU-seconds/day (~5.7 minutes of one core), before entity events and persistence.
  • A 100-node graph’s fixed clock cost is roughly 32 CPU-seconds/day.
  • Therefore: first eliminate irrelevant ticks and repeated setup; do not replace the engine merely because it recomputes.
  • The one-million-node accepted macro shape is a qualitatively different availability risk and must be blocked before optimization work.
  • Add allocation/latency budgets in CI so a compiled executor is justified by measured wins.
Best first energy-saving change: detect whether the compiled graph has any time-dependent descendants. If not, remove the one-second server tick entirely. Then pass changed entity/source IDs and evaluate only their downstream closure.

Recommended deterministic execution architecture

Keep the current full evaluator as a reference oracle. Add a deploy-time compiler and a server executor with explicit transactions. This gives incremental work without adopting glitch-prone stream semantics.

1 · Input batchChanged entity IDs, fetch IDs, graph revision, clock deadline.
2 · SnapshotFreeze one entity/source/time epoch for the transaction.
3 · Dirty closureReverse dependencies identify affected nodes and sinks.
4 · Topological evalEach node runs once and returns every output plus next memory.
5 · Atomic commitPublish output vectors and memory together at one epoch.
6 · Effect executorReconcile committed sink intents through per-sink serialized queues.

Compile once per deploy

  • Decode structure, canonicalize known node schemas, and validate graph semantics.
  • Expand reachable macros under a cumulative budget; reject collisions afterward.
  • Assign dense numeric node/pin indices—no delimiter keys.
  • Build topological order, incoming slots, reverse adjacency, sink list, and entity/source indexes.
  • Annotate node metadata: stateful, time-dependent, effectful, persistence policy, and optional next wake-up.
  • Fingerprint only semantic graph content; unused macro/layout edits must not redeploy.

Transaction invariants

  • All inputs read from one immutable epoch.
  • A node executes at most once per transaction.
  • All outputs of one node are one value vector and publish atomically.
  • State transitions return nextMemory; no mutation during speculative/render evaluation.
  • Diamonds see either the previous epoch or the next epoch—never mixed branches.
  • Pulse outputs are transaction-scoped so incremental caching cannot leave a stale true.
  • Sink intents are derived after stabilization from the same snapshot and included in the commit; external HA effects run afterward and cannot feed back into the same transaction.
  • Stable ordering uses topological index plus a deterministic node-ID tie-break.
Clock policy: specify whether values are continuous wall time, second-quantized, or deadline-driven. For server actuation, schedule the next relevant threshold/deadline where possible; for editor display chips, a separate low-priority visual cadence can update elapsed text.

Follow-up research: compiling time-aware graphs

Verdict: yes—with an important limit. Reactive Wire can stop evaluating the whole graph every second. Event-stable branches can sleep until an entity, fetch, deployment, or configuration event dirties them, and many time predicates can wake at an exact semantic boundary. Raw now, raw elapsed values, and other continuously observable ramps cannot be maintained by one terminal deadline; they need an explicit sampling contract, lazy presentation, or a compatibility cadence restricted to their affected subgraph.

Why this fits the current engine

  • The graph is already an acyclic, typed dataflow shape after macro expansion; expansion produces one flat node/edge graph at shared/engine/expand.ts:57-70.
  • Time is already injected as one epoch-millisecond value, not read ambiently inside nodes, preserving reproducible transactions (shared/engine/evaluate.ts:115-137, shared/engine/nodes/time.ts:17-19).
  • The evaluator already memoizes pins within one full pass and waits until values are materialized before deriving sink calls (shared/engine/evaluate.ts:149-167,268-303).
  • The canonical 1,000-node benchmark is only ~3.94 ms; compilation should retain this evaluator as the from-scratch oracle rather than replace its semantics.

What prevents simply deleting the tick

  • The server and editor independently force a whole evaluation every second (src/server/runtime.ts:164-180,273-281; frontend/src/App.tsx:247-253,271-289).
  • now changes at every clock instant and since is a rounded ramp; neither has a useful final “next change” (shared/engine/nodes/time.ts:17-41).
  • Edge outputs are ordinary cached booleans described as true “for one update.” Removing incidental reevaluation without formal pulse lifetime can leave a stale true visible (shared/engine/nodes/edge.ts:10-30).
  • Node evaluation is per output pin and mutates shared memory, so incremental publication is not generally node-atomic (shared/engine/node-def.ts:77-92; shared/engine/evaluate.ts:158-167,223-265).

Compiled state

  • Stable topological rank and deterministic node-ID tie-break; deployment rejects cycles instead of producing a runtime cycle value.
  • Dense input bindings and reverse adjacency; no delimiter-built runtime identity keys.
  • Observer/necessity roots: deployed sinks and durable state on the server; visible pins/history in preview.
  • Entity indexes for explicit entity nodes, reseed-from-world configuration, and sink target reconciliation.
  • Fetch-node identity, last committed all-output record, semantic equality, memory policy, deadline generation, and pulse transaction.
  • A min-heap ordered by (deadlineEpochMs, topoRank, nodeId, generation); stale generations may wake early but can never suppress an earlier valid wake.

Transaction invariants

  • Each admitted external event has a documented sequence; do not coalesce distinct off→on→off transitions into one snapshot. Deadlines equal at one logical instant may be batched.
  • All dirty dependencies settle before a user; every node steps at most once and returns all outputs plus draft memory.
  • Unchanged semantic output cuts propagation, analogous to Salsa backdating/Jane Street cutoffs.
  • Pulse values are readable only in their creating transaction, including fan-out; no cleanup transaction creates a fabricated falling edge.
  • Sinks derive intents once after graph stabilization from the same HA snapshot. Internal state and intent commit together; HA service calls are external effects and cannot be atomic with memory.
  • Non-ok values retain the current no-actuation rule and cancel any deadline whose proof depended on an unavailable value.

Temporal certificate and threshold arithmetic

A useful internal contract is a conservative nextInvalidationAt: absent external changes, the node promises that its observable output is constant on [transactionNow, nextInvalidationAt). null means no known time-only invalidation; it must not mean “continuously changing.” A stronger exact nextChangeAt can be added only where a node proves it. Pure combinations conservatively take the minimum relevant child certificate; short-circuit logic and select may later suppress masked branches.

type Temporal =
  | { kind: "held" }
  | { kind: "affine"; originMs: number; value: number; ratePerMs: number; quantum?: number };

type NodeStep = {
  outputs: Readonly<Record<string, RuntimeOutput>>; // every pin, one publication
  nextMemory?: NodeMemory;                           // transaction-local draft
  nextInvalidationAt?: number;                       // absolute epoch ms
};

stabilize(event, actualNow):
  apply the event to one immutable input epoch
  mark indexed roots and all valid deadlines <= actualNow dirty
  pop dirty nodes in topological order
  evaluate each once; enqueue users only when semantic output changed or a pulse fired
  derive sink intents from settled values; atomically commit internal state
  arm one bounded host timer for the earliest valid deadline

Do not calculate deadlines from idealized algebra alone. Current elapsed durations round to 0.1 seconds before comparison (shared/engine/engine-support.ts:98-100,137-142), and comparison uses strict JavaScript operators (shared/engine/engine-support.ts:119-131). Thus roundedElapsed >= 10.0 nominally flips near 9.95 seconds, while the existing tested roundedElapsed < 10.0 is false at exactly ten minutes (test/time.test.ts:198-201). Solve for a candidate crossing, then probe neighboring integer milliseconds through the same reference evaluator. Cover increasing/decreasing slopes, < ≤ > ≥ == !=, negative values, ties, and quantization. Equality over an affine value needs an explicit discrete-millisecond or sampling policy.

Scheduler and clock hazards

  • A Node timer is a minimum elapsed delay, not an exact civil-time alarm; callbacks may be late and equal-delay order is not a contract.
  • Native delays above 2,147,483,647 ms (~24.86 days) are unsafe. Chunk long waits and always re-read wall time on wake.
  • Backward clock steps require an early-fire guard and rearm. Forward steps/suspend can make a far deadline late, so retain a bounded scheduler watchdog or platform clock-change signal; it need not evaluate the graph when nothing is due.
  • Drain every deadline <= actualNow in stable order and require the replacement deadline to be strictly later to prevent zero-delay loops.
  • Calendar semantics need an IANA time zone plus explicit DST gap/fold and missed-occurrence policies; persist semantic deadlines, not timer handles.

Current integration gaps

  • EntityFeed.onEntities reports no changed IDs (src/ha/client.ts:18-22); add deltas or diff snapshots once at the boundary.
  • Fetch completion drops the source-node identity into a parameterless callback (src/server/poller.ts:43-48,88-107); pass node ID/version to dirty only its users.
  • Entity maps are implicit dependencies for reseeding and sink reconciliation, not just explicit graph wires (shared/engine/evaluate.ts:220-240,295-302).
  • The editor and server must share the same transaction/deadline semantics. Browser display/history sampling may remain presentation-only (frontend/src/canvas/use-value-history.ts:45-69).
  • Deploy/stop must invalidate deadlines and queued work by generation, extending the lifecycle pattern already at src/server/runtime.ts:189-219.

User experience & design system

Visual system strengths

  • Semantic surfaces, text, accent, health, type colors, and typography are centralized and mapped into Tailwind.
  • Type hues remain stable across aesthetics/modes, making the graph language learnable.
  • Error, unavailable, stale, boolean true/false, and selection use line pattern, opacity, icons, and text—not color alone.
  • Draft/live/disconnected and dry-run/live sink copy is unusually explicit for a home automation tool.
  • Storybook covers many component states and all theme combinations.
  • Palette drawer, inspector sheet, scrim, safe-area padding, and mobile action bar are a good responsive foundation.

System weaknesses

  • Accessibility behavior is not centralized: custom modals, buttons, checkbox, tabs, tooltips, and live messages drift independently.
  • Spacing, radius, control height, motion, and focus are repeated as arbitrary CSS/Tailwind literals rather than tokens/primitives.
  • The runtime hard-codes the IDE aesthetic while Blueprint/Warm exist only in Storybook.
  • Small typography and faint contrast optimize density beyond comfortable readability.
  • No axe/Storybook accessibility addon or keyboard interaction suite protects the design system.
Recommended UX foundation: introduce Button, IconButton, Field, Checkbox, Dialog, Tabs, Tooltip, and live-region primitives with shared focus/motion/touch contracts. Then make canvas keyboard wiring a first-class product flow rather than patching individual handles.

Library choices

Keep React Flow

It is appropriate UI infrastructure and the code uses stable registries, dynamic handle remeasurement, custom edges, and bring-your-own-runtime correctly. Accessibility of custom handles remains the application’s responsibility.

Keep Yjs, fix the model

Yjs is suitable for offline/concurrent editing. The issues come from snapshot projection, whole-entry array/scalar modeling, handshake/ack policy, and duplicated package instances—not from CRDT choice itself.

Keep the HA client

subscribeEntities and callService are the right primitives. Extend the adapter with changed IDs and connection epochs instead of replacing the library.

Do not use RxJS as the core

Per-stream combineLatest does not by itself enforce atomic diamond updates or multi-output node commits. RxJS can be useful at async boundaries, not as the graph’s semantic substrate.

Signals are optional, not the plan

A signal library may implement dirty propagation, but batching, node-wide output vectors, memory transitions, pulses, and sink ordering still need a transaction layer. A compact compiled executor better matches the domain.

Add targeted support libraries

Use Radix Dialog (as already decided) or a robust native dialog primitive; add @storybook/addon-a11y/@axe-core/playwright. A schema library such as Zod/Valibot can decode protocol structure, but custom graph semantic validation is still required.

Package structure: migrate root/frontend to an npm workspace with one lock and one Yjs instance. The current dual installs require explicit resolve.dedupe workarounds in Vite and Vitest (frontend/vite.config.ts:10-13, vitest.workspace.ts:24-27).
